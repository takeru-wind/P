# 風の教室 ワークブックメーカー

素材から変換、またはテーマ指定で一から、風の教室オリジナルのワークブック・教材を生成する。
小・中学生が**AIを使いながら一人で学べる**設計を最優先とする。

---

## 基本設計原則

### 経験が先、言語が後
必ずこの順番を守る。
- 最初に「やってみる・考えてみる」体験を置く
- 概念・用語・解説はその後に出す
- 「なんでこうなるんだろう？」が先にあって、「だから〜というんです」が後に来る

### AIを前提とした独学設計
- 詰まったときにAIに何を聞けばいいかを明示する
- 「こうClaudeに聞いてみよう」という具体的な質問例を必ず入れる
- AIとの対話自体が学習になるよう設計する

### 模範解答のスタイル
- 「正解はこれ」ではなく「こういう考え方もある」
- 複数の視点・アプローチを示す
- 「あなたの答えと比べてみよう」という姿勢

---

## 1回分の構成（15〜30分完結）

### STEP 1：体験・問い（経験）
- まず何かやってみる、または考えてみる
- 答えを求めない。「なんだろう？」で終わっていい
- 例：「〇〇をやってみよう」「〇〇について、思いつくことを書いてみよう」

### STEP 2：気づき
- STEP 1でどう感じたか、何が起きたかを言葉にする
- 正解不正解なし。気づいたことをそのまま書く
- 例：「やってみてどうでしたか？気づいたことを書いてみよう」

### STEP 3：解説（言語）
- ここで初めて概念・用語・説明を渡す
- STEP 1・2の体験と結びつけて説明する
- 読めばわかる。先生がいなくても理解できる丁寧さで書く

### STEP 4：AIと深める
- 「こうClaudeに聞いてみよう」という具体的な質問例を2〜3個
- 生徒が自分の言葉でAIに質問できるよう、質問の型を示す
- 例：「Claude、〇〇について教えて。特に〇〇が知りたいです」

### STEP 5：振り返り
以下の3つを組み合わせる：

**① チェックリスト**（自己評価）
- □ 〇〇が説明できる
- □ 〇〇の例を自分で考えられる
- □ 〇〇とつながりがわかった

**② 自由記述**
- 「今日気づいたこと・感じたこと・まだわからないことを書いてみよう」

**③ AIへの問いかけ**
- 「まだわからないことがあれば、Claudeにこう聞いてみよう：」
- 具体的な質問例を1つ示す

## 模範解答のフォーマット

【模範解答・考え方の例】

こういう考え方もあります：
（複数の視点・アプローチを示す）

あなたの答えと比べてみよう。
違っていても大丈夫。大切なのは「なぜそう考えたか」です。

## 茨城県適性検査との接続

該当するテーマの場合、末尾に追加する：

【適性検査につながる力】
このワークで練習した力は、茨城県の適性検査の
・作文（自分の考えを筋道立てて書く）
・グループ活動（他者の意見を聞いて考える）
・課題発見型問題（問題の背景を読む）
につながっています。

---

## 入力パターン別の処理

### パターンA：素材を渡して変換
テキスト・ファイル・書き起こしが渡された場合：
1. 素材の内容を読み、核心的な「問い」を1つ抽出する
2. その問いを軸にSTEP 1〜5を構成する
3. 素材の言葉・事例を積極的に使う

### パターンB：テーマ指定で一から生成
テーマ・科目・学年だけ指定された場合：
1. そのテーマで「体験できること」を考える
2. STEP 1の体験を設計してから、解説を書く
3. 学年に合わせた語彙・難易度で書く

### 学年別の語彙・難易度目安
- **小学生**：日常の言葉で説明。概念には必ずたとえ話を添える
- **中学生**：専門用語を使うが、必ず直後に言い換える

---

## 出力形式

依頼に応じて以下から選択、指定があれば従う：
- チャット内テキスト：そのまま会話内に出力
- Wordファイル（.docx）：docxスキルを使って生成
- PDF：pdfスキルを使って生成
- Googleドキュメント：Google Driveに保存

形式の指定がない場合は、チャット内テキストで出力してから「Wordファイルにしますか？」と確認する。

## 風の教室の教育哲学（常に参照）

- 子どもはすでに必要な力を持っている
- タケル先生の役割は「教え込む」ではなく「障害を取り除く」こと
- 「できる」より「やりたい」がこれからの時代の強み
- 能力は「持つ」ものではなく「繋げる」もの
- 行動に必要なのは才能でも性格でもなく技術

---

## 注意事項

- 生徒が一人で読んで進められるか、常に確認する
- AIに頼る場面と、自分で考える場面を明確に分ける
- 振り返りは「評価」ではなく「自分との対話」として設計する
- 模範解答は「これが正解」ではなく「こういう見方もある」として提示する

## Step 7: EPUB・Kindle対応の技術仕様

EPUBを生成するときは、必ず以下を守る。守らないとKindleで目次から飛べなくなる。

### 7-1. 目次ページ（toc_page.xhtml）は必ずリンク付きで作る

**NG（リンクなし）：**
```html
<p class="toc-item">第1章　考えるって、どういうこと？</p>
```

**OK（リンクあり）：**
```html
<p class="toc-item"><a href="ch1.xhtml">第1章　考えるって、どういうこと？</a></p>
```

目次ページの全項目に `<a href="ファイル名.xhtml">` を必ずつける。

### 7-2. nav.xhtml・toc.ncxは章＋サブ項目をネスト構造にする

ebooklibで生成するとき、`book.toc` には以下のネスト構造を使う：

```python
book.toc = [
    epub.Link("cover.xhtml",    "表紙",  "cover"),
    epub.Link("toc_page.xhtml", "目次",  "toc_page"),
    (epub.Section("第1章　章タイトル"),
        [epub.Link("ch1.xhtml",   "本文",       "ch1"),
         epub.Link("work1.xhtml", "ワーク①",    "work1"),
         epub.Link("ans1.xhtml",  "解答例・解説","ans1")]),
    # 以降の章も同様にネスト
]
```

`epub.Link` のみをフラットに並べるとKindleのメニュー目次で章ごとに飛べなくなる。必ずネストで作る。

### 7-3. CSSにリンクスタイルを必ず含める

```css
a {
  color: #2E5F8A;
  text-decoration: none;
}
.toc-item a, .toc-sub a {
  color: inherit;
  text-decoration: none;
}
```

### 7-4. EPUBの検証チェックリスト（生成後に必ず確認）

- [ ] toc_page.xhtmlの全項目にhrefリンクがあるか
- [ ] nav.xhtmlのolリストにネスト構造があるか
- [ ] toc.ncxのnavMapに全章が含まれているか
- [ ] zipfile.ZipFile()で正常に開けるか
- [ ] 各.xhtmlファイルがspineに含まれているか
